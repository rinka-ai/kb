## TL;DR

Flue is a strong compact case study for the KB because it makes the "sandbox agent" thesis concrete: a TypeScript agent handler initializes an agent runtime, opens a session, and gives the model a filesystem, shell, tools, skills, command grants, typed outputs, child tasks, and deployment surfaces. The reusable lesson is not just "use sandboxes." It is that a good agent framework needs a portable control plane for context, runs, capabilities, events, and infrastructure lifecycle. Flue has promising primitives, but several current sharp edges show where a better framework should make those contracts more explicit.

## Key Claims

- Flue frames an agent as a headless programmable harness rather than a chat UI or model wrapper.
- The core user API is intentionally small: `init()`, `agent.session()`, then `session.prompt()`, `session.skill()`, `session.task()`, or `session.shell()`.
- The runtime abstracts sandboxes through `SessionEnv`, with implementations for empty virtual just-bash sandboxes, local host-mounted filesystems, Cloudflare virtual/R2-backed filesystems, Cloudflare container sandboxes, and Daytona.
- Built-in tools are filesystem and execution oriented: `read`, `write`, `edit`, `bash`, `grep`, `glob`, and `task`.
- `defineCommand()` is a good capability pattern: privileged host CLIs can be exposed by name while keeping sensitive environment variables out of the model-visible sandbox unless explicitly opted in.
- `session.task()` gives Flue an embedded subagent pattern: child sessions share the sandbox but use separate message history, role overlays, and working-directory-specific context discovery.
- Typed result schemas make prompt and skill calls usable as orchestration steps instead of only conversational text.
- MCP is treated as a trusted-code adapter: the handler connects to a remote MCP server, namespacing discovered tools and passing them into `init()`.
- Flue persists message history as a session DAG with compaction entries, and Cloudflare deployments store sessions in Durable Object SQLite by default.
- This is session persistence, not full durable execution: there is not yet a first-class run object with pause, resume, status, approvals, replay-safe side-effect boundaries, or reconnectable event history.
- The biggest architecture lesson is the context-plane mismatch: roles are bundled at build time, while `AGENTS.md` and skills are discovered from the sandbox filesystem at `init()` time. That means behavior changes depending on whether the target sandbox actually contains those files.
- Open issues reinforce the same maturity gaps: request validation, error envelopes, observability, dev diagnostics, Cloudflare container behavior, Durable Object migration lifecycle, and explicit context loading all need first-class framework semantics.

## Important Details

- `packages/sdk/src/types.ts` defines the main public contracts: `FlueContext`, `AgentInit`, `FlueAgent`, `FlueSession`, `SessionEnv`, `SessionStore`, `ToolDef`, command types, build plugins, and event types.
- `packages/sdk/src/client.ts` creates request-scoped Flue contexts and enforces one `init()` per agent id per request.
- `packages/sdk/src/agent-client.ts` owns agent-level sessions, session deletion, shell execution, task session creation, and per-agent command/tool inheritance.
- `packages/sdk/src/session.ts` is the core runtime wrapper around `@mariozechner/pi-agent-core`. It handles prompt/skill/task/shell calls, scoped runtime mutation, role/model resolution, custom tools, child tasks, compaction, history persistence, events, and result extraction retry.
- `packages/sdk/src/session-history.ts` stores messages, compactions, and branch summaries as a parent-linked entry graph with a single active leaf.
- `packages/sdk/src/context.ts` loads `AGENTS.md`, `CLAUDE.md`, and `.agents/skills/<name>/SKILL.md` from the session environment's current working directory.
- `packages/sdk/src/agent.ts` implements the built-in tools. Its `grep` and `glob` wrappers are simple shell commands rather than a rich indexed retrieval layer.
- `packages/sdk/src/result.ts` uses delimiter-based result extraction with a follow-up extraction prompt when the model omits markers.
- `packages/sdk/src/compaction.ts` summarizes older conversation history, tracks read/modified file paths from assistant tool calls, and supports threshold and overflow-triggered compaction.
- `packages/sdk/src/build.ts` discovers agent files and roles, parses `triggers` with regex, writes `dist/manifest.json`, and delegates entry generation to a build plugin.
- `packages/sdk/src/build-plugin-node.ts` emits a Hono server with sync, SSE, and webhook modes. Node session storage defaults to process-memory unless a custom `SessionStore` is provided.
- `packages/sdk/src/build-plugin-cloudflare.ts` emits a Worker plus one Durable Object class per webhook agent. It uses Cloudflare Agents SDK fibers for webhook keepalive and DO SQLite for session persistence.
- `packages/sdk/src/cloudflare-wrangler-merge.ts` is a reminder that deployment lifecycle is part of the framework: Wrangler config merging, DO bindings, migrations, sandbox binding detection, and deploy redirect files are all framework-owned responsibilities.
- `packages/sdk/src/dev.ts` is careful about Node versus Cloudflare reload models, but open issues show that build/dev error visibility still needs stronger invariants.
- `packages/connectors/src/daytona.ts` shows the connector shape: adapt an existing sandbox SDK into `SandboxFactory` instead of owning the provider's lifecycle.
- The website and deploy docs expose the intended positioning: Flue wants to be "Astro or Next.js, but for agents," with write-once deploy targets for Node.js, Cloudflare, GitHub Actions, GitLab CI/CD, and future hosts.

## My Notes

- Flue is probably the cleanest small TypeScript example in the KB of the "agent as programmable sandbox harness" pattern.
- The most useful thing to copy is its API ergonomics: `init` plus session methods are easy to explain, and typed prompt results make agent calls compose with normal application code.
- The most useful thing not to copy blindly is the split between build-time and runtime context discovery. In a stronger framework, project context should be a single portable context plane with explicit sources, scope, freshness, and target compatibility.
- Flue also makes the "durability is not just storage" lesson concrete. A Cloudflare Durable Object session store is valuable, but a robust long-running agent framework still needs run status, pause/resume, approval interrupts, replay discipline, event history, and side-effect boundaries.
- The open issues are unusually informative because they reveal what users actually feel when the runtime contract is underspecified: silent context absence, silent dev wedges, permissive HTTP invocation, migration failures, weak observability, and confusing platform-specific behavior.

## Open Questions

- Should the KB split "agent definition surface" from "agent runtime surface" more explicitly? Flue makes that distinction visible: agent files and roles are build artifacts, but skills and `AGENTS.md` currently behave like runtime filesystem artifacts.
- What is the minimum durable-run contract for a sandbox-agent framework: run id, status, event log, resumable operation, approval pause, cancellation, retry policy, and artifact manifest?
- Should sandbox agents expose a standard capability ledger that records commands, tools, env grants, network access, files read, and files modified across a run?
- How much target-specific infrastructure should a framework own before it becomes too platform-shaped to stay portable?
- Can a better framework preserve Flue's tiny TypeScript ergonomics while adding explicit request validation, typed payloads/env, observability, and context-plane guarantees?

## Source Text

Selected source text and code anchors inspected:

- README tagline: Flue is "The Agent Harness Framework" and is positioned as a TypeScript framework for headless, programmable agent harnesses.
- `README.md`: Flue says it is not another AI SDK, but a runtime-agnostic framework for deploying agents to Node.js, Cloudflare, GitHub Actions, GitLab CI/CD, and similar targets.
- `packages/sdk/src/types.ts`: `SessionEnv` is the universal sandbox interface; `FlueSession` exposes `prompt`, `shell`, `skill`, `task`, and `delete`; `FlueEvent` defines streaming events such as tool start/end, command start/end, task start/end, compaction start/end, idle, and error.
- `packages/sdk/src/context.ts`: `AGENTS.md`, `CLAUDE.md`, and `.agents/skills/` are read from the session environment at runtime.
- `packages/sdk/src/build.ts`: roles and agent trigger metadata are discovered from the host workspace at build time, while `AGENTS.md` and skills are explicitly not bundled.
- `packages/sdk/src/build-plugin-cloudflare.ts`: Cloudflare builds emit Durable Object classes for webhook agents, use DO SQLite for session storage, and rely on the Cloudflare Agents SDK for routing and fibers.
- GitHub issue #16 documents the runtime-context mismatch: skills and `AGENTS.md` silently do not load on the default Cloudflare virtual sandbox because the sandbox filesystem starts empty.
- GitHub issue #18 documents a framework-contract gap in request validation and error envelopes: non-POST requests and malformed JSON can still invoke agents in current behavior.
- GitHub issue #19 identifies typed payload/env and response metadata as likely next DX/API improvements.
- GitHub issue #7 tracks observability gaps, especially parity between CLI and Cloudflare logging.
