## TL;DR

Cognee is valuable to this KB as implementation evidence for the claim that memory, retrieval, session state, skills, evaluation, and self-improvement are converging into one runtime/control-plane layer. The code supports permanent graph memory, session cache memory, trace capture, multiple graph/vector retrievers, eval adapters, feedback-weighted graph updates, skill ingestion into graph-backed `Skill` nodes, recording `SkillRunEntry` objects through `cognee.remember`, and generating reviewable `SkillImprovementProposal` nodes from low-scoring runs. The repository supports the broad direction of the tweet, though the exact `SkillChangeEvent` class named in the tweet was not found in the inspected branch.

## Key Claims

- Cognee presents itself as an open-source memory control plane for agents, not just a vector-store wrapper.
- Its core public API now frames memory around `remember`, `recall`, `forget`, and `improve`.
- Skills can be ingested explicitly as dataset-scoped graph nodes rather than treated only as files outside the memory system.
- Skill execution feedback can be stored through the same `remember` API shape used for other typed memory entries.
- Skill improvement is proposal-first: the system can generate a graph-backed proposal from failure evidence, and applying it is a separate operation.
- Session memory, graph memory, agent traces, and skill runs are implemented as related but distinct lanes inside the same project.
- `cognify`, `search`, `recall`, and `improve` are the deeper control-plane loop: build graph memory, retrieve through explicit modes, route across session/trace/graph state, then promote feedback and session traces into durable graph memory.
- The retrieval layer exposes many search types rather than one "RAG" path, including chunks, lexical chunks, graph completion, graph summary completion, context extension, chain-of-thought, temporal, Cypher, natural-language, coding-rules, and agentic completion.
- The repo includes eval infrastructure that closely matches the paper: HotPotQA, TwoWikiMultiHop, MuSiQue adapters, DeepEval and direct-LLM graders, exact match/F1 metrics, bootstrap confidence intervals, and dashboards.

## Important Details

- `README.md` describes Cognee as a memory control plane combining embeddings, graphs, and cognitive-science-inspired approaches.
- `README.md` shows `await cognee.remember(...)` for permanent graph memory and session memory, plus `await cognee.recall(...)` for retrieval.
- `cognee/skill.md` is itself a Cognee skill, showing that the project dogfoods skill-style instructions for Cognee-specific workflows.
- `cognee/api/v1/cognify/cognify.py` implements the graph-building side: document classification, chunk extraction, LLM graph extraction/summarization, optional ontology configuration, optional temporal graph extraction, and vector/graph storage.
- `cognee/modules/search/types/SearchType.py` exposes the retrieval taxonomy as a first-class enum rather than hiding it behind one generic query API.
- `cognee/modules/search/methods/get_search_type_retriever_instance.py` maps each `SearchType` to a retriever and its relevant parameters, including `wide_search_top_k`, `triplet_distance_penalty`, `feedback_influence`, `neighborhood_depth`, and global-context-index settings.
- `GraphCompletionRetriever` retrieves graph triplets by combining vector search over node/edge collections with graph projection and top-triplet importance scoring.
- `brute_force_triplet_search` searches vector collections such as entity names, text summaries, entity types, document chunks, and edge relationship names, then maps vector distances back onto graph nodes/edges.
- `CompletionRetriever` is the conventional chunk-RAG path, retrieving `DocumentChunk_text` results and passing the chunk text into the LLM.
- `AgenticRetriever` extends graph completion with a ReAct-style loop over memory, skills, and tools. It loads only skill names/descriptions first, exposes `load_skill` for progressive disclosure, records opened skills as `SkillRun` nodes, and stores session Q&A when caching is enabled.
- Built-in agentic tools include `memory_search` and `load_skill`; tool execution is permission-gated by dataset/user scope.
- `cognee/api/v1/remember/remember.py` routes typed `MemoryEntry` objects, including `SkillRunEntry`, before the normal add/cognify path.
- `remember(..., content_type="skills")` explicitly ingests `SKILL.md` files as dataset-scoped `Skill` nodes; the code intentionally does not auto-detect skill paths.
- `Skill` is a `DataPoint` with name, description, procedure, declared tools, content hash, dataset scope, and embeddable `search_text`.
- `SkillRunEntry` records selected skill, task text, result summary, score, feedback, error data, candidate skills, router version, and tool trace.
- `SkillRun` persists the selected skill, candidate skills, score, feedback, tool trace, session id, and error fields.
- `SkillImprovementProposal` stores old procedure, proposed procedure, runs used, confidence, rationale, and status.
- `improve_skill_from_config` and `improve_skill` generate proposals from low-scoring or errored `SkillRun` records, then require an explicit apply path to mutate the stored skill procedure.
- The demo at `examples/demos/skill_feedback_loop/skill_feedback_loop_demo.py` mirrors the tweet's code: ingest skills, run an agentic completion, record a weak skill run, generate a proposal, and apply it.
- `cognee.api.v1.recall` can search session QA, agent traces, graph context, and graph retrieval lanes; auto routing uses a rule-based classifier over query patterns.
- `cognee.api.v1.improve` runs staged memory maintenance: apply feedback weights, persist session Q&A, persist agent trace steps, run default memify enrichment, optionally build a global context index, and sync graph knowledge back to session cache.
- Feedback weights are applied only to graph elements actually referenced by prior retrieval, through `used_graph_element_ids`.
- `cognee.modules.agent_memory.agent_memory` wraps async agent functions with memory retrieval, session-memory retrieval, and trace persistence. It enforces dataset read/write scope for graph-backed memory.
- The agent-memory runtime derives a memory query from a fixed string, a method parameter, or the first usable text argument; retrieved context is capped before entering the wrapped agent.
- Unit tests confirm dataset-scoped skill IDs, explicit agentic retrieval mode, progressive skill loading, dataset-scoped skill-run persistence, and proposal-first mutation.
- Integration tests confirm that session traces can be stored, graph memory can be retrieved from shared datasets with permissions, and session memory works without dataset permissions when graph retrieval is disabled.
- `eval_framework` implements benchmark adapters for HotPotQA, TwoWikiMultiHop, MuSiQue, dummy tasks, BEAM, and logistics-system simulations; evaluation includes exact match, F1, context coverage, DeepEval, direct LLM grading, bootstrap intervals, and dashboard generation.
- `evals/README.md` describes comparative QA evaluation against Mem0, Graphiti, and LightRAG, but the claims there should be treated cautiously unless raw results and environment parity are inspected.
- A search of the inspected branch found no exact `SkillChangeEvent` symbol; the implementation evidence is better described as graph-backed skill/run/proposal nodes rather than a verified event-emission class.

## My Notes

- Cognee is a stronger source than the tweet because the code shows the actual design boundary: memory and skills share storage and feedback infrastructure, but they remain typed artifacts with explicit routing and apply steps.
- The useful pattern for this KB is not "collapse everything into memory"; it is "store procedural artifacts, run evidence, and improvement proposals in one inspectable graph so the maintenance loop can see across them."
- The proposal-first apply path is important governance. It avoids treating a failed run as automatic permission to rewrite a reusable skill.
- The exact API surface is still moving, so the KB should cite inspected commit `8b0d687` for implementation claims.
- This source supports updating [[agent-memory]], [[agent-skills]], [[agent-harnesses]], and [[rag]], but it does not by itself justify a new `world-models` concept page.
- The better near-term KB upgrade is to frame "memory control plane" as an [[agent-memory]] pattern: explicit lifecycle verbs, multiple memory lanes, retrieval mode routing, feedback promotion, eval fixtures, and guarded mutation of procedural artifacts.
- Cognee's use of `used_graph_element_ids` is worth stealing conceptually: feedback should update the evidence that actually contributed to an answer, not the whole corpus or whatever happened to match later.
- The global context index is a cousin of this KB's concept pages: it creates higher-level summaries over graph summaries so retrieval can start from a compact map before drilling into local triplets.
- The code also reinforces a caution: one control plane can accumulate many surfaces quickly. Without clear typed lanes and evals, "memory" risks becoming an overloaded bucket for sessions, traces, graph facts, skills, tool telemetry, and generated summaries.

## Open Questions

- Should skills, memory entries, and improvement proposals share one retrieval namespace, or should they share one graph while preserving typed retrieval lanes?
- What eval threshold and review workflow should be required before a skill-improvement proposal is applied?
- How should a memory system expose skill routing without turning every stored fact into a possible instruction?
- What is the right audit trail when a skill procedure changes because of memory-derived evidence?
- How should feedback weights decay or revert when later answers show that previously boosted graph elements were misleading?
- Should global context summaries be built from graph summaries, source documents, or evaluated answer traces?
- When should session traces be promoted as semantic graph memory versus kept as operational history only?
