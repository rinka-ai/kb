## TL;DR

Aya is a strong internal-codebase source because it shows how a small AI product can stay disciplined while integrating voice, telephony, email, search, LLM fallback, dashboards, queues, auth, and tenant data. The reusable pattern is not the real-estate domain itself; it is the way the repo keeps the agent core provider-neutral, makes request/session/run state durable, pushes side effects through gateways and an outbox, and forces coding agents to honor product scope and existing conventions before adding abstractions.

## Key Claims

- Aya is an AI-first platform for Portuguese real-estate agencies; V1 is intentionally one inbound voice receptionist flow, not a general CRM or multi-agent platform.
- The repo uses a Bun/Turborepo TypeScript monorepo with `apps/api`, `apps/web`, `apps/worker`, `apps/scripts`, and shared `packages/*`.
- Source-of-truth docs carry product and architecture decisions: `docs/PRODUCT.md`, `docs/ARCHITECTURE.md`, `docs/DECISIONS.md`, `docs/DESIGN.md`, and `docs/WEB_DATA_ROUTING.md`.
- `AGENTS.md` is intentionally short and points agents to `CLAUDE.md`, where failure-mode-specific coding rules live.
- `packages/core` is the provider-neutral runtime layer. It depends on Aya-owned interfaces and domain types, not provider SDKs or database row types.
- Gateway packages isolate provider integrations: `@aya/gateway-voice-call`, `@aya/gateway-email`, `@aya/gateway-http`, and the channel-agnostic `@aya/gateways` interface package.
- The core request model separates business state (`Request`) from the durable session log and from individual agent runs.
- BullMQ and Redis locks implement per-request serialization: concurrent events for one request queue behind `lock:request:{requestId}` while different requests can proceed in parallel.
- External delivery is outbox-driven. Broker email delivery is emitted as durable intent and drained by a worker rather than called inline from webhooks or agent runs.
- Voice architecture is structured-first. ElevenLabs runs live STT/LLM/TTS and returns post-call structured outputs; Aya validates those outputs and uses its own model provider only for repair/synthesis fallback.
- Mid-call voice tools are read-only, tenant-scoped, and HMAC/callback-auth verified. Tenant is resolved from the provisioned agent id, not from caller-controlled body fields.
- Constants and branded IDs are a major convention: enum-shaped values, lifecycle states, channels, outcomes, event types, and role keys are named constants rather than scattered literals.
- The web dashboard follows a server-seed plus React Query cache pattern for repeated operational workbenches, with `router.refresh()` reserved for global server-context changes.
- Design discipline is explicit: UI work must use `docs/DESIGN.md`, token roles, bilingual EN/PT copy parity, no brand/vendor names in operator-facing strings, and screenshot review for HTML/CSS changes.
- The shared AI research KB is a development-time architecture resource only; Aya's runtime knowledge surface is tenant-owned agency content, not the internal dev KB.

## Important Details

- Root `package.json` names the workspace `aya`, uses `bun@1.1.34`, and exposes `dev`, `build`, `lint`, `typecheck`, `test`, DB, reset, ElevenLabs provisioning/test, and cleanup scripts.
- `README.md` describes the top-level shape: `apps/api` as Hono server, `apps/web` as the operator dashboard, `apps/worker` as BullMQ consumer process, and packages for core, gateways, DB, config, logger, and types.
- `CLAUDE.md` defines the stack: Bun, Turborepo, strict TypeScript, Hono, Postgres/Drizzle/pgvector, BullMQ/Redis, Better Auth, Resend, Telnyx, ElevenLabs Conversational AI, OpenRouter, OpenAI embeddings, S3 on Railway, Pino, Biome, Zod, Railway, EU region, EN/PT.
- `tsconfig.base.json` is strict and maps `@aya/*` aliases to package entrypoints and subpaths.
- `biome.json` enforces formatting and important lints: unused imports/variables, no committed `console.log`, `useImportType`, `useExportType`, and warnings for explicit `any` and non-null assertions.
- `turbo.json` exposes required environment variables as task pass-through/env config while keeping dev non-cached and build/typecheck/test dependency-aware.
- `docs/ARCHITECTURE.md` states the core boundary directly: the agent core knows no channels or providers; gateways translate provider events into normalized envelopes and proposed actions into provider API calls.
- `docs/DECISIONS.md` keeps an ADR log covering Bun/Turbo/Hono, ElevenLabs + Telnyx, Resend, Postgres/Drizzle/pgvector, BullMQ/Redis locks, outbox, Better Auth, Railway, OpenRouter, OpenAI embeddings, single V1 flow, bilingual support, multi-tenancy, channel-agnostic outcomes, and dev-only KB usage.
- `apps/api/src/app.ts` is the HTTP composition root: CORS, auth, admin/tenant surfaces, provider webhook signature middleware, tool routes, and webhooks are wired centrally.
- `apps/api/src/deps.ts` defines the dependency envelope passed into app/routes: DB, logger, object storage, embeddings, queues, auth, cache, email gateway, and public app URL.
- `packages/core/src/queue/process-request-worker.ts` mirrors DB helper types locally so `@aya/core` remains free of `@aya/db` imports.
- `packages/core/src/queue/outbox-drain-loop.ts` owns the delivery lifecycle as a named pipeline: claim, build, send, classify, persist, retry.
- `packages/types/src/constants.ts` is deliberately zod-free so client components can import constants without dragging the Zod runtime into the dashboard bundle.
- `packages/types/src/index.ts` reintroduces Zod schemas for server-side boundaries while re-exporting constants and branded domain types.
- `docs/WEB_DATA_ROUTING.md` records a concrete performance lesson: workbench lists should use server-rendered seeds and React Query rather than repeated no-store RSC refreshes.
- `docs/DESIGN.md` keeps a strong design posture: "Monet vibes, operator-grade software" means calm, useful, tokenized operational UI rather than decorative AI/SaaS tropes.

## My Notes

- Aya is especially useful as evidence for a small, serious AI product architecture: the LLM is important, but deterministic state, boundary schemas, provider adapters, outbox retries, and tenant scoping carry the product.
- The repo avoids the common "agent platform" trap. It says no to multi-agent orchestration, runtime MCP exposure, custom eval platforms, prebuilt future modules, and speculative package extraction until a real flow needs them.
- Its coding-agent instructions are good because they are not a generic prompt manifesto. They encode concrete failures: dependency sprawl, parent-relative imports, zod leaking into client bundles, raw enum strings, duplicated constants, hidden dashboard latency, missing bilingual strings, unrendered UI changes.
- The distinction between dev-time shared KB and runtime agency content is worth preserving. It prevents a dangerous ambiguity around "the KB" and keeps tenant data, architecture research, and agent tool context from collapsing into one surface.
- The architecture uses local mirrors of DB helper signatures in core. This is a useful boundary tactic, but it carries maintenance cost: if DB helper semantics change, mirror types must be updated intentionally.
- Aya's dashboard conventions are a compact version of the Uniswap UI lesson: UI quality comes from source architecture, not just component appearance.

## Open Questions

- When should Aya promote internal `packages/core` modules into separate packages, and what evidence should be required before extraction?
- Should the outbox drain loop add a durable `lease_token` column before multi-process drains become common?
- How much of the voice-agent structured extraction contract should be verified against provider simulations versus real-call traces?
- Should the dev-time KB reference in `CLAUDE.md` be mirrored into a repo-local summarized note so future agents can work if the hosted MCP is unavailable?
- How should Aya evaluate whether tenant-specific prompts and voice profiles are improving outcomes without creating a custom eval platform too early?
