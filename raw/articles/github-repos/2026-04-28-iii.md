## TL;DR

iii is a concrete implementation of the "harness as backend" pattern: backend systems and agent harnesses are modeled through the same primitives. A function does work, a trigger causes work to run, and a worker connects functions and triggers to the engine. Agents, API services, queue consumers, cron jobs, state reactions, observability handlers, browser tabs, and sandboxed execution environments can all be treated as workers or triggerable functions.

## Key Claims

- Backend complexity can be collapsed into three primitives: Function, Trigger, and Worker.
- The same function can be invoked directly, exposed through HTTP, scheduled by cron, triggered from state changes, or placed onto a queue without changing the underlying business logic.
- Workers can be written in Node.js, Python, Rust, or browser environments and connect to the engine over WebSocket.
- Live discovery matters because functions and triggers registered by one worker become visible and usable across the system in real time.
- Live extensibility matters because new workers can add capabilities to a running system without a full redeploy or a separate service registry.
- Live observability matters because tracing, metrics, logs, alerts, and trace-tree queries are treated as engine-level capabilities rather than application-specific glue.
- Sandboxes are exposed through the same trigger mechanism, so isolated execution becomes another backend capability instead of a separate agent-infrastructure category.

## Important Details

- The README defines iii as a single engine with Function, Trigger, and Worker as its core abstraction.
- The docs describe built-in workers for HTTP, queue, cron, state, stream, pub/sub, observability, exec, bridge, and sandbox capabilities.
- External workers connect over WebSocket, register functions and triggers, and can run locally, in containers, on separate hosts, or in browser tabs.
- Worker lifecycle includes automatic registration, deregistration, reconnect, and re-registration behavior.
- Function IDs use a namespace convention such as `orders::process`, and any registered function can be triggered across language and service boundaries.
- Trigger actions support synchronous calls, fire-and-forget calls, and enqueueing for async processing.
- The observability worker uses OpenTelemetry-style traces, metrics, logs, alert rules, sampling, and built-in query functions such as `engine::traces::list` and `engine::traces::tree`.
- The sandbox API exposes microVM lifecycle and filesystem operations through triggerable functions like `sandbox::create`, `sandbox::exec`, and `sandbox::stop`.
- The repo ships agent skills for iii primitives and architecture patterns, including agentic backends, workflow orchestration, queue processing, state reactions, and observability.

## My Notes

- The useful synthesis is not "iii is another framework"; it is "the harness/backend boundary may disappear when agents and services share the same execution substrate."
