## TL;DR

Goose is a useful implementation exemplar for the KB because it assembles many of the abstractions discussed elsewhere here into one working stack: a provider-agnostic core agent loop, a unified extension runtime for built-in and MCP tools, persistent sessions and scheduling, explicit approval and inspection layers, and multiple client surfaces that reuse the same underlying engine.

## Key Claims

- Goose positions itself as a local general-purpose AI agent with desktop, CLI, and API surfaces rather than as a single-interface coding tool.
- The repository is organized around a Rust core crate (`crates/goose`) with separate CLI, server, ACP, MCP, and SDK layers built on top.
- Provider integration is normalized through a shared `Provider` trait and registry rather than hard-coded to one model vendor.
- Tooling is unified through an extension runtime that can host built-in platform extensions as well as MCP servers over stdio and streamable HTTP.
- The implementation treats durable session state, scheduling, cancellation, replay, and resumability as first-class runtime behaviors.
- Goose also demonstrates how skills can stay discoverable without loading full skill bodies into the hot path until needed.

## Important Details

- The workspace centers on `crates/goose`, with `goose-cli`, `goose-server`, `goose-acp`, `goose-mcp`, and `goose-sdk` as adjacent crates and a separate `ui/` workspace for Electron, ACP text UI, and SDK packaging.
- The main agent orchestration in `crates/goose/src/agents/agent.rs` combines provider calls, prompt management, extension loading, tool routing, retry logic, compaction, and approval handling.
- `crates/goose/src/providers/base.rs` defines a common provider contract, while `providers/init.rs` wires a large registry of built-in and declarative providers.
- `crates/goose/src/agents/extension_manager.rs` manages both local built-ins and external MCP clients, including stdio and streamable HTTP transports, OAuth fallback, timeout handling, and resource access.
- `crates/goose/src/agents/platform_extensions/mod.rs` registers built-in extensions such as `developer`, `analyze`, `skills`, `summon`, `todo`, `apps`, and `chatrecall`, showing a mixed model of deterministic tools plus LLM-mediated capabilities.
- `crates/goose/src/session/session_manager.rs`, `execution/manager.rs`, and `scheduler.rs` show that Goose treats session identity, background work, cancellation tokens, and cron-like scheduling as core runtime concerns rather than bolt-ons.
- The server routes and SSE replay flow in `crates/goose-server/src/routes/reply.rs` and `session_events.rs` show a concrete pattern for reconnectable client event streams around long-running agent work.
- Goose includes a security and approval pipeline with security, egress, adversary, permission, and repetition inspection layers before tool execution, which makes runtime policy explicit instead of leaving it in prompt text.
- The repo is broad and somewhat transitional: the Electron desktop app is active, there is a newer Tauri-based `goose2` path, and the project also exposes ACP and SDK surfaces.

## My Notes

- Goose is most valuable in this KB as a practical harness and runtime case study, not as a canonical theoretical source.
- It reinforces the KB's distinction between model intelligence and harness responsibility: most of the durable value is in the provider layer, extension runtime, session store, event surfaces, and governance hooks around the model loop.
- The repo is also a useful reminder that product breadth creates architectural drag. Goose shows both the upside of shared primitives and the complexity cost of supporting many providers, runtimes, and UI surfaces at once.

A native desktop app for macOS, Linux, and Windows. A full CLI for terminal workflows. An API to embed it anywhere. Built in Rust for performance and portability.
