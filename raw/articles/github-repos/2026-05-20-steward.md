## TL;DR

Steward belongs in the KB because it is not just a crypto wallet repo; it is agent action infrastructure. The reusable pattern is a control plane that keeps wallet keys and API credentials out of agent runtimes, routes signing and API access through policy gates, supports approval queues, and records audit evidence. The strongest caveat is also documented by the project itself: current policy enforcement is an API-path invariant before the vault signs, not an unbypassable cryptographic signer boundary.

## Key Claims

- Agent wallet infrastructure should be treated as part of agent security, not as a separate fintech side topic.
- Autonomous agents need scoped custody surfaces for wallet keys, API keys, database credentials, and paid API access.
- Steward's architecture combines wallet custody, secret custody, auth, policy evaluation, proxy routing, audit logs, and SDK/UI/plugin surfaces.
- The project supports EVM and Solana wallets, with new agents receiving both chain-family keypairs in the inspected code.
- Policy evaluation supports spending limits, approved addresses, rate limits, time windows, auto-approval thresholds, allowed chains, and newer reputation-based policy types in the schema/evaluators.
- Auto-approval thresholds turn high-risk signing attempts into pending approvals rather than immediate signatures.
- The proxy gateway is the broader lesson for agent tools: agents can call external services through a broker that injects credentials and logs access, instead of holding real API keys.
- Multi-tenant identity, tenant-scoped agents, tenant-scoped secrets, session auth, JWTs, passkeys, OAuth, SIWE, and refresh-token rotation make Steward closer to a managed agent action layer than a single wallet library.
- Steward includes an ElizaOS plugin, showing the agent-framework integration path: sign, transfer, check spend, list approvals, and expose wallet/balance providers.
- The project's security docs explicitly narrow the marketing claim: v1 policies run in API route handlers before vault calls; the vault primitive itself does not enforce policies.

## Important Details

- `README.md` presents Steward as auth and wallet infrastructure for autonomous agents, with a vault, policy engine, auth, proxy gateway, and audit log.
- `VISION.md` frames the mission around agents managing real money, paid APIs, and credentials without relying on plaintext `.env` secrets.
- `docs/concepts/architecture.mdx` describes Steward as a chokepoint between agents and the outside world where API calls, transactions, and secret access can be policy-checked, logged, and metered.
- `docs/security/threat-model.mdx` is unusually useful because it documents security preconditions, adversary classes, and places where older product phrasing overclaims the implementation.
- `docs/adrs/0004-policy-enforcement-model.mdx` clarifies that policies are currently called from API route handlers before invoking the vault. This is a meaningful protection against prompt-injected agents but not a separate cryptographic enforcement layer.
- `packages/policy-engine/src/engine.ts` approves only when all hard policies pass and either the auto-approve threshold passes or no threshold blocks. If hard policies pass but the threshold fails, the engine returns `requiresManualApproval`.
- The same engine currently treats an empty policy set as auto-approved, while some docs describe default deny. The threat model notes this as dangerous but valid for testing, so KB synthesis should preserve the caveat.
- `packages/api/src/routes/vault.ts` stores pending, rejected, signed, and broadcast transaction records and writes approval-queue rows when evaluation requires manual approval.
- `packages/vault/src/vault.ts` creates both EVM and Solana keys for new agents, encrypts them, stores public addresses separately, and decrypts keys only for signing operations.
- `packages/vault/src/secret-vault.ts` stores encrypted secrets and returns metadata for listing/get paths; plaintext secret values are for internal credential injection.
- `packages/proxy/src/handlers/proxy.ts` resolves target routes, matches tenant-scoped secret routes, checks proxy rate/spend limits, decrypts credentials, injects them into outbound requests, forwards to the target API, records audit entries, and tracks spend for known LLM APIs.
- The proxy strips inbound auth and hop-by-hop headers before injecting credentials into outbound requests.
- `packages/db/src/schema.ts` shows tenant, agent, wallet, encrypted key, policy, transaction, approval, secret-route, and audit-log structures, which makes the control-plane shape explicit.
- `packages/sdk/src/client.ts` exposes the agent-facing API for wallet creation, signing, policies, history, addresses, RPC passthrough, secrets, routes, approvals, audit, and dashboard-style surfaces.
- `packages/eliza-plugin/src/index.ts` packages Steward as ElizaOS actions, providers, evaluators, and a service.
- Self-hosting and embedded PGLite modes matter for agent infrastructure because local CLI/desktop agents often need custody without delegating all secrets to hosted SaaS.

## My Notes

- This source sharpens the second-brain boundary: include adjacent infrastructure when it becomes an agent control surface. Wallet custody, paid API access, credential proxying, auth, and signing approvals are all inside the AI-era systems boundary even when they look like fintech or auth at first glance.
- Steward is most useful to the KB as a concrete pattern for "agent wallets" meaning policy-bounded authority, not merely blockchain addresses.
- The proxy gateway may be the most generally reusable idea: for agent systems, credential handling should move from environment variables into brokered, logged, policy-aware network paths.
- The project also demonstrates why source inspection matters. Product-facing pages say policy is enforced at the signing layer, while the ADR and threat model correctly say the current enforcement boundary is the API path before vault calls.
- The empty-policy auto-approve behavior is a major operational caveat. An agent wallet control plane is only as strong as its default policy posture and provisioning workflow.
- The future signer-process design sketched in ADR-0004 is the more robust architecture: keep the master signing secret out of the API process and require policy attestations before signing.
- The KB should treat Steward as evidence for a new concept, [[agent-wallet-infrastructure]], while also feeding [[agent-security]], [[agent-tools]], [[managed-agents]], [[payment-integrations]], and [[agent-harnesses]].

## Open Questions

- Should a production agent-wallet system default to deny when no policy exists, even if tests and demos become less convenient?
- What is the right type-level or process-level invariant to force `evaluate -> sign` ordering?
- How should policy attestations be represented when a separate signer process verifies approvals?
- Should agent wallet infrastructure expose one general spending policy model across on-chain transactions, paid API calls, and SaaS metering?
- How should audit logs distinguish model intent, tool-call request, policy result, human approval, and final external side effect?
- What portability layer should connect wallet-control planes to agent frameworks without giving each framework its own signing semantics?
