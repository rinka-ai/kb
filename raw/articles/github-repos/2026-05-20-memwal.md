## TL;DR

MemWal belongs in the KB because it is agent memory infrastructure: encrypted, owner-scoped, semantically searchable memory with SDK, MCP, OpenClaw plugin, relayer, smart-contract, indexer, and restore surfaces. Its strongest contribution is the user-owned memory pattern: memory spaces are scoped by owner, namespace, and package/deployment; access is granted through onchain delegate keys; encrypted blobs live on Walrus; the vector index is rebuildable. Its strongest caveat is equally important: the default relayer path sees plaintext memory during embedding/encryption/decryption, so privacy depends on choosing the right trust posture.

## Key Claims

- Agent memory should be portable and user-owned, not locked inside one platform's chat database.
- MemWal models long-term memory as `remember`, `recall`, `analyze`, `ask`, and `restore` operations over an owner-scoped memory space.
- Memory spaces are bounded by owner address, namespace, and MemWal package ID/deployment, which separates personal, work, research, and app-specific memory.
- The default SDK path gives developers a Web2-friendly API while the relayer handles embedding, SEAL encryption/decryption, Walrus upload/download, vector search, and restore.
- The manual SDK path shifts embedding and SEAL operations to the client so the relayer only sees encrypted payloads, vectors, and blob IDs.
- Access control is split: ownership and delegate authorization live onchain in Sui; relayer-side execution handles encryption, indexing, rate limits, quotas, and search.
- Delegate keys let agents and apps access memory without handing over the owner wallet. Each account supports up to 20 delegate keys in the inspected contract/docs.
- Walrus is treated as the durable encrypted source of truth, while PostgreSQL/pgvector is the fast rebuildable semantic index.
- Restore is a first-class operation: if the index is lost or incomplete, the relayer can rediscover blobs, decrypt, re-embed, and re-index a namespace.
- The MCP package makes encrypted memory callable from Codex, Claude Desktop, Claude Code, Cursor, and similar clients through six tools plus inline browser login.
- The OpenClaw/NemoClaw plugin shows a harness integration pattern: auto-recall before prompt assembly, auto-capture after turns, explicit search/store tools, namespace isolation, and prompt-injection filtering.
- The system is beta and actively evolving, so KB claims should preserve the current trust boundaries rather than treating the project as settled infrastructure.

## Important Details

- `README.md` describes MemWal as a privacy-first AI memory layer for encrypted memories on Walrus with semantic search.
- `SKILL.md` frames MemWal as a TypeScript SDK for persistent encrypted memory, decentralized storage, onchain ownership, delegate access, semantic recall, Vercel AI SDK middleware, and OpenClaw/NemoClaw plugin integration.
- `packages/sdk/package.json` lists `@mysten-incubation/memwal` version `0.0.4`; `packages/mcp/package.json` lists `@mysten-incubation/memwal-mcp` version `0.0.1`; `packages/openclaw-memory-memwal/package.json` lists `@mysten-incubation/oc-memwal` version `0.0.2`.
- `docs/fundamentals/architecture/core-components.md` defines six core components: SDK, relayer, MemWal smart contract, indexer, Walrus, and indexed database.
- The relayer verifies key authorization against the contract, generates embeddings, encrypts/decrypts through SEAL, uploads/downloads Walrus blobs, stores/searches vectors in PostgreSQL, and scopes operations by owner plus namespace.
- `docs/fundamentals/architecture/data-flow-security-model.md` explicitly says onchain ownership/delegate authorization is cryptographic, while the relayer is trusted for plaintext in the default path.
- Trust posture options are managed relayer, self-hosted relayer, TEE relayer pattern, and manual client flow.
- `docs/fundamentals/architecture/how-storage-works.md` says `remember` creates an async job, embeds plaintext, encrypts with SEAL, uploads encrypted blob to Walrus, and indexes vector metadata in PostgreSQL/pgvector.
- Recall embeds the query, searches pgvector by owner plus namespace, downloads matching encrypted blobs, decrypts them, and returns plaintext sorted by distance.
- Restore queries onchain Walrus blob objects by owner and namespace, compares against the local DB, downloads missing blobs, decrypts, re-embeds, and inserts vector entries.
- `docs/fundamentals/concepts/memory-space.md` defines the memory-space boundary as `owner + namespace + app_id`.
- `docs/contract/delegate-key-management.md` documents delegate generation, owner-only onchain registration, revocation, a 20-key limit, and account deactivation as a kill switch.
- `services/contract/sources/account.move` shows `MemWalAccount`, `DelegateKey`, version gating, owner checks, delegate-key limits, account active state, and `seal_approve` policy semantics.
- `services/server/src/auth.rs` verifies Ed25519 signatures over timestamp, method, path/query, body hash, nonce, and account ID; timestamps have a five-minute window and nonces are checked through Redis.
- `services/server/src/rate_limit.rs` implements weighted per-account, per-delegate-key, and storage-quota limits.
- `services/server/src/routes/remember.rs` stores original plaintext encrypted but summarizes large text before embedding so long documents stay retrievable.
- `services/server/src/routes/recall.rs` caches recall query embeddings, caps recall limits, surfaces dropped entries, and can rerank hydrated results.
- `services/server/src/services/ranker.rs` supports semantic plus recency scoring while preserving default cosine order when recency is not enabled.
- `docs/mcp/overview.md` and `docs/mcp/how-it-works.md` describe six tools: `memwal_remember`, `memwal_recall`, `memwal_analyze`, `memwal_restore`, `memwal_login`, and `memwal_logout`.
- The MCP package stores local credentials in `~/.memwal/credentials.json`, intercepts login/logout locally, and bridges stdio MCP traffic to the relayer after authentication.
- The MCP login flow includes loopback-only callback binding, origin/host checks, a state token, and a 5-minute window; logout deletes the local credential file but does not revoke the onchain delegate key.
- `packages/openclaw-memory-memwal` integrates through hooks and tools: before-prompt auto-recall, after-turn auto-capture, `memory_search`, and `memory_store`.
- The OpenClaw plugin filters prompt-injection-looking memories, HTML-escapes recalled memory blocks, strips recalled-memory tags before capture, and scopes namespaces from the session key.

## My Notes

- MemWal is exactly the kind of adjacent repo that should live in the broader second-brain version of this KB. It is not an AI model, but it is infrastructure for agent memory, privacy, ownership, retrieval, and cross-client persistence.
- The durable lesson is the split between encrypted source of truth and rebuildable search index. Walrus stores the encrypted memory; pgvector makes it usable; restore ties the two back together.
- The ownership story is stronger than a normal vector database because access is tied to account and delegate keys, not just app database rows. But the privacy story still depends on relayer choice.
- The default relayer path is good developer experience but not end-to-end private from the relayer operator. The manual path is the important high-trust alternative because plaintext and SEAL operations remain client-side.
- The MCP package is worth tracking because it turns memory into a cross-client tool surface. A memory saved from one MCP host can be recalled in another if they share the same MemWal account and namespace.
- The OpenClaw plugin demonstrates good harness hygiene: auto-recall and auto-capture are separate from explicit tools, namespace scoping is injected even on recall failures, and stored memories are treated as possible prompt injection.

## Open Questions

- Should default AI memory middleware ever auto-save without a user-visible review queue, or should it only propose memories?
- How should users reason about vector embeddings as sensitive data even when plaintext blobs are encrypted?
- What governance should exist around delegate-key creation, rotation, revocation, and per-agent namespace scope?
- Should cross-client memory use a shared default namespace, or should each client and project scope memory explicitly?
- When is a TEE relayer good enough, and what client UX verifies enclave identity before sending plaintext?
- How should memory restore preserve freshness, deletion intent, and supersession semantics rather than simply rebuilding every discoverable blob?
- What is the right blend of semantic similarity, recency, importance, and keyword/entity signals when memory text is encrypted behind SEAL?
